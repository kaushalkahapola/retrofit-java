KEY FINDINGS
================================================================================

LINE NUMBER HANDLING:
  - Internal: 0-indexed (for slicing)
  - Output: 1-indexed (for human display)
  - Always convert with `index + 1` when returning to user
  - Use saturating arithmetic: `saturating_add()`, `saturating_sub()`

FILE EDITING STRATEGY:
  - EXACT string matching ONLY (no fuzzy matching)
  - Use `replace()` for replace-all
  - Use `replacen(..., 1)` for first occurrence only
  - Validate BEFORE mutating the file
  - Return comprehensive output for verification

PATCH GENERATION:
  - Single hunk for entire file (not multi-hunk)
  - oldStart and newStart always = 1
  - Lines prefixed with "-" (removed) or "+" (added)
  - For visualization only, not for application

ERROR HANDLING:
  - Errors are converted to messages (not exceptions)
  - Pre-condition validation (inputs)
  - Existence checks (file/string found)
  - Safe arithmetic (no panic on boundaries)

OUTPUT COMPREHENSIVENESS:
  - Always return original file content
  - Include structured patch for visualization
  - Echo parameters back (old_string, new_string, etc.)
  - Include metadata (file paths, line counts)
  - Include provenance flags (user_modified)

================================================================================
IMPLEMENTATION PATTERNS
================================================================================

1. Validation Before Mutation
   - Check inputs first
   - Then write to disk
   - Return original state for audit

2. Dual Indexing
   - Internal: 0-indexed arrays
   - Output: 1-indexed for human display
   - Consistent conversion pattern

3. Exact String Matching
   - No context matching
   - No fuzzy matching
   - Deterministic and auditable

4. Comprehensive Output
   - Include verification data
   - Echo parameters back
   - Provide metadata

5. Boundary Safe Arithmetic
   - Always use saturating operations
   - Check bounds before slicing
   - Use min() for limits

6. Error Message Conversion
   - Convert Err to message string
   - Include is_error flag
   - Report to user as message

================================================================================
WHAT CLAW INTENTIONALLY DOES NOT IMPLEMENT
================================================================================

✗ Fuzzy/context matching for edits
  Reason: Deterministic and auditable only

✗ Patch application (only generation)
  Reason: Simple visualization-only approach

✗ File locking or transactions
  Reason: Direct writes for simplicity

✗ Rollback mechanism
  Reason: Changes are immediate

✗ Merge conflict resolution
  Reason: Simple string replacement only

✗ Automatic retry on failure
  Reason: Errors are reported to user

✗ Line-based patching
  Reason: Full file replacement approach

These are intentional design choices, not oversights.
