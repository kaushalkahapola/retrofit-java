## Problem Statement

### Initial Issue
The evaluation workflow was sending **all hunks from .java files** to the agentic generation system, including non-code content like SQL, YAML, JSON, XML strings embedded in source code. Agents shouldn't regenerate non-code content.

### Secondary Issue (Addressed Later)
The system was also sending **auto-generated Java files** (like ANTLR-generated Lexer/Parser files) to agents for modification. These files are automatically generated from grammar files (.g4) and should never be regenerated by agents - they have massive line number changes and cascading effects.

**Example**: The issue.patch from Elasticsearch contained:
- `EsqlBaseLexer.java` - 2184+ line diff (ANTLR auto-generated)
- `EsqlBaseParser.java` - 128+ line diff (ANTLR auto-generated)

These were being sent to agents when they should be applied directly as auxiliary hunks.

## Key Design Decisions

1. **Two-tier detection**: Combine filename patterns (for auto-generated) + content patterns (for non-code)
   - Reason: Auto-generated files can be detected efficiently by filename; non-code content needs hunk inspection

2. **Whole-hunk approach**: If a hunk contains ANY non-code content, skip entire hunk from agents
   - Reason: Conservative - avoid breaking non-code patterns

3. **Pattern-based detection**: Uses regex patterns, not file extensions
   - Reason: Detects non-code content WITHIN .java files

4. **Graceful degradation**: Invalid regex patterns are skipped, detection continues
   - Reason: Robustness against pattern errors

5. **Per-hunk filtering**: Tests each hunk individually
   - Reason: Same file can have mixed content hunks

6. **Early exit optimization**: Check auto-generated status once per file (not per-hunk)
   - Reason: Performance - don't repeat expensive filename checks

## Known Limitations & Future Improvements

1. **Listener/Visitor patterns may have false positives**
   - Currently: `*Listener.java` and `*Visitor.java` are detected as auto-generated
   - Risk: User-created classes named SomeListener/SomeVisitor would be skipped
   - Solution: Could add file content inspection to verify ANTLR markers

2. **Protobuf patterns are conservative**
   - Matching `*Pb.java` is broad and may catch non-protobuf files
   - Solution: Could inspect file headers for protobuf-generated markers

3. **No detection of file content headers**
   - Some tools add "// This file is automatically generated" comments
   - Could enhance by scanning file headers for generation markers
   - Not yet implemented due to performance considerations
