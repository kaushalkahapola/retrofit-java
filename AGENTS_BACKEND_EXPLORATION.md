# Agents-Backend Codebase Exploration: Comprehensive Analysis

**Date**: April 3, 2026  
**Focus**: Understanding Phase 3 failure detection and recovery mechanisms for semantic adaptation implementation

## EXECUTIVE SUMMARY

The agents-backend pipeline implements a 4-agent retry loop system where **Phase 3 (Hunk Generator/File Editor)** applies adapted code hunks to target files. The current architecture uses:

1. **Deterministic text-matching** for anchor resolution with **no semantic understanding**
2. **LLM-driven hunk generation** that can hallucinate context mismatches
3. **Limited error context** at failure points (mainly "anchor not found" messages)
4. **Validation feedback loop** that re-routes to earlier agents for retry

The codebase has **extensive file-reading APIs** and **structural analysis utilities** that can be leveraged for semantic adaptation, but they're not currently integrated into failure recovery.

## PHASE 3 ARCHITECTURE

### Entry Point: `file_editor_node()` (agents/file_editor.py)
**Lines 728-1174**

The node orchestrates:
1. Reset file to HEAD (clean state)
2. Apply str_replace edits deterministically
3. Capture diff via `git diff HEAD`
4. Intent verification (LLM check)
5. Reset file to HEAD again (restore clean state)

**Key Contract**: 
- Edits succeed deterministically OR fail with explicit anchor resolution attempts
- No LLM is used to *generate* hunks anymore (replaced by planning agent)
- All diffs are **mechanically generated by git** (machine-correct)

### Alternative: Hunk Generator (agents/hunk_generator.py - Legacy)
**Lines 1353-2000+ (partial, file has 3000+ lines)**

The older approach where:
- LLM rewrites hunks directly to diff format
- Higher hallucination risk
- Used when planning agent not available

## RETRY MECHANISMS

### 1. **Validation Loop** (graph.py:68-163)
```
Attempt 1 → validation fails
    ↓
Analyze failure_category:
    - "generation_contract_failed" → replan (planning_agent)
    - "context_mismatch" → replan (planning_agent)
    - "path_or_file_operation" → remap (structural_locator)
    - default → retry generation (hunk_generator with error context)
    ↓
Attempt 2 (with error context injected)
    ↓
Attempt 3 (final attempt)
    ↓
MAX_VALIDATION_ATTEMPTS reached → END
```

### 2. **Deterministic Anchor Retry** (file_editor.py:409-492)
```
Primary attempt: _apply_edit_deterministically()
    ↓ (if old_resolution_failed:not_found)
    ↓
Secondary attempt: _retry_with_file_check()
    - Grep for longest non-empty line in old_string
    - Use grep result as anchor point
    - Reconstruct line window around that point
    - Try replacement again with reconstructed window
```

### 3. **Hunk Structure Stabilization** (hunk_generator.py:959-994)
```
LLM generates candidate hunk
    ↓
_stabilize_hunk_structure(base_hunk, candidate_hunk)
    - Preserve base structure (context/removed lines)
    - Replace only '+' lines with candidate '+' lines
    - Fallback to base if '+' count mismatch
```

## WHAT'S MISSING FOR SEMANTIC ADAPTATION

### Gap 1: No Semantic Analysis at Failure Point
Currently:
- Error: "anchor text not found"
- No analysis of *where* the code actually is
- No signal about structural divergence

**Available to leverage**:
- `grep_in_file()` (find where anchor actually is)
- `read_file_window()` (see what's actually there)
- Method fingerprinter (detect if method was renamed/refactored)
- Structural matcher (compare class structure similarity)

### Gap 2: Limited Context About Divergence
Currently:
- Validation returns error message + raw tool output
- No structured analysis of what changed
- No semantic similarity scores

**Available to leverage**:
- AST-based fingerprinting (method structure vector)
- Call-graph analysis (Jaccard similarity of dependencies)
- Inheritance chain matching (superclass/interface comparison)

### Gap 3: No Fallback Code Analysis for Failed Anchors
Currently:
- If anchor not found → retry with whitespace tolerance
- If still fails → give up, ask for replan

**Available to leverage**:
- Full file parsing via `get_class_context()`
- Symbol location via `find_symbol_locations()`
- History tracing via `git_pickaxe()`
- These are phase 2 tools but NOT used in phase 3 retry

### Gap 4: No Integration of Phase 2 Mapping Confidence
Currently:
- Phase 2 returns `anchor_confidence` (high/medium/low)
- Phase 3 ignores this signal
- Same retry strategy regardless of mapping quality

**Should leverage**:
- Route low-confidence mappings to different retry (remap first)
- Use `anchor_reason` to decide between "try harder" vs "remap"
- Pre-alert Phase 3 about high-divergence files

### Point 5: Semantic Tooling in Phase 3
**Location**: file_editor.py (if we keep it) or hunk_generator.py

Add toolkit access for:
1. `get_class_context()` - extract method body from target
2. `grep_repo()` - search entire codebase for snippet
3. `git_pickaxe()` - trace where code moved to
4. Method fingerprinter - match by structure

Workflow:
```
Anchor resolution fails
    ↓
Call method fingerprinter on target file
    - Get all method signatures + call graphs
    - Score against original method
    - If high score match found: update insertion_line, retry
    ↓
If fingerprinter finds no match:
    Call grep_repo(snippet) to search entire codebase
    - If found elsewhere: semantic location change detected
    - Return structured error: "code moved to different file"
    ↓
If still nothing:
    Call git_pickaxe(snippet) to trace history
    - Detect: renamed, refactored, deleted
    - Return structured error with history context
```

## SUMMARY OF KEY FINDINGS

1. **Current State**: Deterministic anchor matching + whitespace tolerance + retry with error context
   - No semantic understanding of divergence
   - Limited failure diagnostics
   - Retry blindly without structural analysis

2. **Failure Detection Capability**: 
   - Excellent at knowing *when* anchors fail (exact match → trimmed → window)
   - Poor at knowing *why* they fail (refactoring? rename? moved code?)
   - No integration with mapping confidence from Phase 2

3. **Available Semantic Tools**:
   - Method fingerprinter (4-tier matching algorithm)
   - Structural matcher (class-level analysis)
   - File context APIs (read windows, grep, line ranges)
   - Code history tools (git pickaxe, blame, log follow)
   - Full-repo search (grep_repo)

4. **Injection Points**:
   - **Phase 3 failure recovery** (before giving up, try semantic analysis)
   - **Phase 2→3 interface** (propagate mapping confidence)
   - **Retry routing** (use semantic signals to choose remap vs. retry)
   - **Error enrichment** (add semantic diagnostics to validation errors)

5. **Low-Hanging Fruit**:
   - Integrate MethodFingerprinter into Phase 3 anchor failure handling
   - Propagate `anchor_confidence` from Phase 2 to Phase 3
   - Enhance validation errors with structural similarity scores
   - Route low-confidence + failed-anchor → structural_locator (remap)
